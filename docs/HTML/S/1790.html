   1 \chapter{Creating new widgets and dialogues}
   2 
   3 This chapter dives into the area of creating your own widgets or dialogues. For
   4 this example we use a real widget and a dialogue under development. The widget
   5 being developed is the progress bar and the screen the initial load screen.
   6 
   7 First we make the widget, then the screen in order to test the widget. It
   8 happens often that, when you need to add a new widget, you need it for a new
   9 dialogue - so you need to create both for testing. Another option is to add the
  10 widget as dummy item to another existing dialogue. When adding a dialogue for
  11 which all widgets already exist this doesn't matter and you can directly dive
  12 into generating the dialogue.
  13 
  14 \section{Creating the widget}
  15 \label{sec:creating_the_widget}
  16 
  17 The widget is normally split in 7 different files. Three .cpp/.hpp files and one
  18 WML file. Usually I start to create the new files by simply copy pasting the old
  19 files and pick a rather small class like timage or something. Then, I start to
  20 update the build system files, followed by creating the code for the
  21 classes\footnote{Obviously I already thought about the design of the class
  22 before that, but that's of less importance in this tutorial.}.
  23 
  24 Now, we start to explain what the various files do. First, I do a general overview of
  25 the files after which I dive more into the implementation of each file.
  26 
  27 \begin{description}
  28 
  29 \item[src/gui/auxiliary/widget\_definition/progress\_bar.*]
  30         These files contain the definition of the progress bar. This mainly involves
  31         the fixed fields for the widget. This widget has no extra fields so the copy
  32         pasted version suffices. This file contains the \emph{static} properties for
  33         a certain definition of a progress bar.
  34 
  35 \begin{description}
  36 \item[hpp] \Liref{widget_definition.hpp} contains the sample code. 
  37         \begin{description}
  38         \item[\Lref{widget_definition.hpp:control}] Normally a
  39                 widget definition inherits from tcontrol\_definition which defines the
  40                 basic mandatory fields for a widget definition and a templated load
  41                 function for the resolution. Most widgets don't add more members to this
  42                 class.
  43         \item[\Lref{widget_definition.hpp:resolution}] Normally a resolution
  44                 definition inherits from tresolution\_definition\_ which define the
  45                 basic mandatory fields for a resolution definition. Not all widgets use
  46                 these members, but most do. Most widgets don't add more members to this
  47                 class, except most container classes that add a grid builder
  48                 definition. Another example is are the scrollbars which define some
  49                 minimum sizes for parts of their control.
  50         \end{description}
  51 
  52 \item[cpp] \Liref{widget_definition.cpp} contains the sample code. 
  53         \begin{description}
  54         \item[\Lref{widget_definition.cpp:textdomain}] Every file in the library
  55                 is inside the ``wesnoth-lib'' text domain.
  56         \item[\Lref{widget_definition.cpp:constructor}] The constructor used is
  57                 pretty typical for all widgets.
  58         \item[\Lref{widget_definition.cpp:resolution_constructor}] The
  59                 constructor used is semi-typical. Resolutions that have their own
  60                 members initialize them in the constructor, but also validate mandatory
  61                 fields for their existence.
  62 
  63                 Also the progress bar only has one state, most widgets have more and
  64                 thus add more states.
  65 
  66                 And maybe the most important part of this constructor is the wiki
  67                 comment. This comment is used to generate wiki pages about the widget.
  68                 These wiki pages are used by content creators to know how to create a
  69                 proper WML file for the class.
  70         \end{description}
  71 \end{description}
  72 
  73 \item[data/gui/default/widget/progress\_bar\_default.cfg]
  74         Now that the widget is defined, we can decide how it should look. Which is
  75         done in this file. Note that in the formula we use ``percentage'' which is
  76         the percentage of the bar to be filled.
  77 
  78         \begin{description}
  79         \item[\Lref{progress_bar.cfg:textdomain}] Every file in the library
  80                 is inside the ``wesnoth-lib'' text domain.
  81 
  82         \item[\Lref{progress_bar.cfg:class}]
  83                 Beginning of the progress bar. This part is much better documented in
  84                 the wiki\footnote{\url{http://wiki.wesnoth.org/GUIWidgetDefinitionWML}}. 
  85                 Since this part is aimed at WML authors, duplicating that effort here
  86                 makes no sense so just visit the wiki.
  87 
  88         \end{description}
  89 
  90 \item[src/gui/auxiliary/window\_builder/progress\_bar.*]
  91         These files contain the code to build a widget from the definition described
  92         above and a config object as defined in the window. These files contain also the
  93         \emph{dynamic} properties of the progress bar. Since the progress bar has no
  94         dynamic content this file is also rather short.
  95 
  96 \begin{description}
  97 \item[hpp] \Liref{window_builder.hpp} contains the sample code. 
  98         \begin{description}
  99         \item[\Lref{window_builder.hpp:control}] Normally a
 100                 window builder inherits from tbuilder\_control which defines the
 101                 basic mandatory fields for a window builder.
 102 
 103                 The function also declares the build function.
 104 
 105                 Widget define their own member if they want to have some runtime
 106                 settings, e.g. the spacer defines its fixed size if needed.
 107         \end{description}
 108 
 109 \item[cpp] \Liref{window_builder.cpp} contains the sample code. 
 110         \begin{description}
 111         \item[\Lref{window_builder.cpp:textdomain}] Every file in the library
 112                 is inside the ``wesnoth-lib'' text domain.
 113         \item[\Lref{window_builder.cpp:constructor}] The constructor used is
 114                 pretty typical for all widgets.
 115 
 116                 If the constructor initializes custom members it may also add some validation.
 117                 This is also true when the widget has a grid, like the listbox or
 118                 multi\_page.
 119         \item[\Lref{window_builder.cpp:build}]
 120                 This build() is semi-typical, it creates the widget, initializes the
 121                 default fields and returns the created object.
 122 
 123                 Widgets with their own members initialize them after initializing the
 124                 default members.
 125         \item[\Lref{window_builder.cpp:wiki}]
 126                 At the end of the file it contains two wiki comment sections:
 127 
 128                 The first defines a macro with a short description of the widget.
 129 
 130                 The second describes the extra fields for the instance of the widget.
 131         \end{description}
 132 \end{description}
 133 
 134 \item[src/gui/widgets/progress\_bar.*]
 135         These files contain the interaction the widget has with the outer world and
 136         how it reacts to events. Again, the progress bar is rather boring. The
 137         interesting part is the setter of the percentage. The value of the
 138         percentage used in the drawing routines is set here\footnote{Other classes have a
 139         separate update canvas, but that's because the canvas in those classes is
 140         invalidated in several functions.}.
 141 
 142 \begin{description}
 143 \item[hpp] \Liref{progress_bar.hpp} contains the sample code. 
 144         \begin{description}
 145         \item[\Lref{progress_bar.hpp:class}]
 146                 Most simple widgets derive from tcontrol, which is the base class of the
 147                 visual widgets. Another common base is tcontainer or
 148                 tscrollbar\_container.
 149 
 150         \item[\Lref{progress_bar.hpp:constructor}]
 151                 The class calls the control constructor with the number of states, which
 152                 is the number of canvases used. Other then that, it does the
 153                 initialization of its members. Some constructors do a bit more
 154                 processing, but where a lot needs to be done it's often deferred to a
 155                 finalize() function.    
 156 
 157         \item[\Lref{progress_bar.hpp:inherited}]
 158                 This section implements or overrides member functions of the base class.
 159                 Some functions are purely virtual and need to be overridden by others to
 160                 change the behaviour.\footnote{Of course, adding here a list of functions which are
 161                 common would be nice. However, the library is still too
 162                 volatile to do so - no need to create a soon out of date list.}
 163 
 164         \item[\Lref{progress_bar.hpp:settersgetters}]
 165                 This section contains the setters and getters for the class. The
 166                 list below shows the common convention where T is the type of the
 167                 setter/getter and ``foo'' the name of the member without trailing
 168                 underscore.
 169 
 170                 \begin{description}
 171                 \item{setter}
 172                         Signature void set\_foo(const T foo); The type T can be by reference
 173                         if a larger class.
 174 
 175                 \item{getter}
 176                         Signature T get\_foo() const { return foo\_; } The type T is always
 177                         returned by value.
 178 
 179                 \item{reference}
 180                         Signature T\& foo() { return foo\_; } This version returns the value
 181                         by reference, and it should also have a const version. These
 182                         functions or at least the non-const version are often not public.
 183 
 184                 \item{pointer}
 185                         Signature T* foo() { return foo\_; } Almost the same as the
 186                         reference version and is used for members that are pointers. The
 187                         text of the reference also applies here.
 188                 \end{description}
 189 
 190                 The setter is normally available. The getter is often omitted when a
 191                 reference or pointer function are used. Especially it there is a const
 192                 version of these functions.
 193 
 194         \item[\Lref{progress_bar.hpp:state}]
 195                 The state has the list of states available and normally has COUNT as
 196                 last value, which in turn is used in the constructor to initialize the
 197                 base class.
 198 
 199         \end{description}
 200 
 201 \item[cpp] \Liref{progress_bar.cpp} contains the sample code. 
 202         \begin{description}
 203         \item[\Lref{progress_bar.cpp:textdomain}] Every file in the library
 204                 is inside the ``wesnoth-lib'' text domain.
 205 
 206         \item[\Lref{progress_bar.cpp:logheader}]
 207                 The common headers used for logging, this makes sure the output of the
 208                 items always have the same format. Used by
 209                 LOG\_FOO \textless\textless{} LOG\_HEADER \textless\textless{} "foo.\textbackslash{}n"; or
 210                 log\_scope(foo, LOG\_SCOPE); % check function signature
 211 
 212         \item[\Lref{progress_bar.cpp:register}]
 213                 Registers a widget, this part is really volatile and still has some
 214                 hoops and rings to jump through. Normally, it works. If not, too bad -
 215                 try to look through the other widgets to see how to fix it. 
 216 
 217                 \textbf{Caveat emptor} Just when all compiles fine and you suddenly get a runtime
 218                 error about your new nice class, the builder claims your widget doesn't
 219                 exist. \emph{Don't panic}, probably you did nothing wrong. Grep TRY in
 220                 src/gui/auxiliary/window\_builder.cpp (at the moment of writing line 96)
 221                 and read the comment above, duly add your class to the list and be
 222                 happy.
 223 
 224         \item[\Lref{progress_bar.cpp:get_control_type}]
 225                 A simple helper to get the human readable name of the class. The first
 226                 version was inlined in the header, but that seems to be an ODR
 227                 violation. At least at some point a compiler or linker complained, can't
 228                 really remember which one. % TODO git pick-axe
 229 
 230         \end{description}
 231 \end{description}

 258 \section{Creating the window}
 259 \label{sec:creating_the_window}
 260 
 261 The window is normally split in 3 files. One .cpp/.hpp file and one WML file.
 262 The dialogue used is the unit attack dialogue, at least an initial version. This
 263 version barely does what it needs to do, and the in game version will be more
 264 polished. This simple version is nice as an example here since it's not too large.
 265 
 266 Now, let's explain what the various files do. First, a general overview of
 267 the files after which I dive more into the implementation of the file.
 268 
 269 \begin{description}
 270 
 271 \item[src/gui/dialogs/unit\_attack.*pp]
 272         These files contain the code of the dialogue. This code `binds' the WML
 273         code and allows the C++ code to show the dialogue show the right data at the
 274         right place. In several cases this also allows the designer of the WML file
 275         to decide which widget s/he picks.
 276 
 277 \begin{description}
 278 \item[hpp] \Liref{unit_attack.hpp} contains the sample code. 
 279         \begin{description}
 280         \item[\Lref{unit_attack.hpp:class}]
 281                 Defines the class itself, which normally inherits from tdialog.
 282                 Sometimes a class inherits from another class, but this other class then
 283 derives from tdialog; for example the wml\_dialogs have a common base and
 284 separate classes for left and right.
 285 
 286         \item[\Lref{unit_attack.hpp:constructor}]
 287                 Dialogues are often created and then shown once before being destroyed,
 288 therefore the constructor often takes all the parameters needed. Since the show()
 289 function shouldn't be overloaded, the only other alternative would be member
 290 functions to set the members. Note that when keeping `references' to widgets these
 291 members must be pointers and set to NULL in the constructor, this due to the
 292 simple fact that the widgets aren't yet created at this point. Also keep in mind that
 293 when the dialogue is shown twice (using the \emph{same} object) the widgets won't be
 294 the same. The window owning the widgets is destroyed after being shown and
 295 a new one created upon the next call to show().
 296 
 297         \item[\Lref{unit_attack.hpp:settersgetters}]
 298                 As said before, there are often no setters, since all is set in the
 299 contructor. Most members also remain hidden, since the caller has the
 300 information. Only members that change are `exported'. The most common example in
 301 a class with a listbox, the last selected item is made public, since this is
 302 often used as the choice the user made.
 303 
 304         \item[\Lref{unit_attack.hpp:window_id}]
 305                 This function is a part of the window registration and only needs to be
 306 declared in the header, the definition will be provided by a macro.
 307 
 308         \item[\Lref{unit_attack.hpp:pre_show}]
 309                 This function is called after the window is created, but before being
 310 shown. This is the point to set the members that point to a widget. As said
 311 before, every time the dialogue is shown a newly created window is shown as well, so
 312 setting the pointers can be unconditional. This is also the place to fill the
 313 widgets with the proper content, if needed. E.g. filling the languages in the
 314 language selection dialogue.
 315 
 316         \item[\Lref{unit_attack.hpp:post_show}]
 317                 Called after the dialogue is shown, this allows you to update certain
 318 statuses. E.g. if the dialogue is closed with the OK button set the selected
 319 language to the newly selected language. You can also update it unconditionally
 320 and let the caller test the status.
 321 
 322         \item[\Lref{unit_attack.hpp:members}]
 323                 At this point the list of private members start.
 324 
 325         \end{description}
 326 
 327 \item[cpp] \Liref{unit_attack.cpp} contains the sample code. Note some
 328 parts are already described more verbosely in the header and thus not mentioned
 329 here.
 330         \begin{description}
 331         \item[\Lref{unit_attack.cpp:textdomain}]
 332                 Like written for the widget, every file is inside the `wesnoth-lib' text
 333 domain.
 334 
 335         \item[\Lref{unit_attack.cpp:wiki}]
 336                 Directly after the includes (inside the gui2 namespace) the wiki
 337 documentation starts.
 338 
 339         \item[\Lref{unit_attack.cpp:register}]
 340                 After the wiki documentation the registration code follows, it defines
 341 the window\_id function and does some other registration parts.
 342 
 343         \item[\Lref{unit_attack.cpp:pre_show}]
 344                 The pre\_show is already mentioned in the header file and it the one
 345 used here uses several static helper functions to do it's job.
 346 
 347         \end{description}
 348 
 349 \end{description}
 350 
 351 \item[src/gui/dialogs/unit\_attack.cfg]
 352         This file contains the WML that defines the widget. It should honour the
 353 restrains set in the wiki comment like \lref{unit_attack.cpp:wiki}. These
 354 constrains set the minimum set of needed widgets and their types, and mentions
 355 the optional widgets. How the widgets are placed and whether only the optional
 356 ones are set is up to the designer of the dialog. The gui toolkit wiki has more
 357 information on the subject. Still some parts are worth mentioning.
 358 
 359         \begin{description}
 360         \item[\Lref{unit_attack.cfg:textdomain}]
 361                 Every file in the library is inside the ``wesnoth-lib'' text domain.
 362 
 363         \item[\Lref{unit_attack.cfg:defines}]
 364                 With larger dialogs, I often first design them on paper and then divide the
 365 dialog in several sections. These sections often are turned into defines in
 366 order to keep the main dialog rather simple. WML tends to be verbose and deeply
 367 nested, and the GUI's also tend to have a lot of nested items, these powers
 368 combined result in deeply nested structure, where one can easily lose the way.
 369 Also note that all local macros are prefixed with \_GUI, this is to avoid clashes
 370 with other macros. The BIG in the names are just because I want to make a
 371 different definition of high resolution screens (this will be the first screen
 372 using that feature, but that will be added later).
 373 
 374         \item[\Lref{unit_attack.cfg:window}]
 375                 Not much to tell about this part, since it's all documented in the wiki. %TODO add ref?
 376 But do note that due to the defines above, the grid in the window itself is
 377 rather simple.
 378 
 379         \item[\Lref{unit_attack.cfg:undefines}]
 380                 The local macros are no longer needed so undefine them, I normally
 381 undefine them in the oposite order of the definitions. I'm quite sure that's not
 382 actually needed, but it feels the right way\texttrademark.
 383 
 384         \end{description}
 385 
 386 \end{description}
