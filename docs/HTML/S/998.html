  31 /**
  32  * Container associating units to locations.
  33  * An indirection location -> underlying_id -> unit ensures that iterators
  34  * stay valid even if WML modifies or moves units on the fly. They even stay
  35  * valid if a unit is erased from the map and another unit with the same
  36  * underlying id is inserted in the map.  In other words it is a doubly indexed ordered map
  37  * with persistent iterators (that never invalidate)
  38 
  39  @note The unit_map is implemented as 2 maps, one unordered map that stores iterators from the ordered map of reference counted pointers to units.
  40  The unordered map provides O(1) find times. The ordered map ensures ordering of units by underlying_id.
  41  The reference counting is what guarantees the persistent iterators.
  42  Storing an iterator prevents only that dead unit's id-map entry from being recovered.
  43 
  44 @note Prefered usages for tight loops follows.
  45 Use the std::pair<iterator, bool> format which checks the preconditions and returns
  46 false in the bool to indicate failure with no change to the unit_map.  true indicates success and the new iterator is in first.
  47 Storing the result iterator prevents the old iterator from entering the fallback recovery code.
  48 This is faster than the old methodology of find to check if empty, insert and then find to check for success.
  49 It is the same method as std::map uses, the C++ way.
  50 
  51 unit_iterator i;  //results are stored here
  52 
  53 Add
  54 std::pair<unit_iterator, bool> try_add(units->add(loc, unit));
  55 if(try_add.second){i = try_add.first;}
  56 
  57 Move
  58 std::pair<unit_iterator, bool> try_add(units->move(src, dst));
  59 if(try_add.second){i = try_add.first;}
  60 
  61 Insert
  62 std::pair<unit_iterator, bool> try_add(units->insert(unitp));
  63 if(try_add.second){i = try_add.first;}
  64 
  65 Replace  (preferred over erase and then add)
  66 std::pair<unit_iterator, bool> try_add(units->move(loc, unit));
  67 if(try_add.second){i = try_add.first;}
  68 
  69 
  70 
  71 
  72  @note The previous implementation was 2 binary tree based maps one the location map pointing to the other.
  73  Lookups were O(2*log(N)) and O(log(N)).  Order was implicit in the id map choosen as the base.
  74  Persistence was provided by reference counting all iterators collectively and only recovering space when
  75  there were no iterators outstanding.  Even 1 iterator being stored caused a leak, because
  76  all space for dead units is not recovered.
  77 
  78  * @note Units are owned by the container.
  79  * @note The indirection does not involve map lookups whenever an iterator
  80  *       is dereferenced, it just causes a pointer indirection. The downside
  81  *       is that incrementing iterator is not O(1).
  82  * @note The code does not involve any magic, so units moved while being
  83  *       iterated upon may be skipped or visited twice.
  84  * @note Iterators prevent ghost units from being collected. So they should
  85  *       never be stored into data structures, as it will cause slowdowns!
  86  * @note By popular demand iterators are effectively permanent. They are handles and not iterators.
  87  * Any access might cause a full lookup. Keeping iterators around holds onto memory.
  88  */
