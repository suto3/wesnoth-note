  41 \section{Prerequisites}
  42 A quick list before getting into details:
  43 \begin{enumerate}
  44 \item Java 6
  45 \item Python 2.x
  46 \item Wesnoth 1.9.x, trunk or newer
  47 \item Eclipse 3.7 ( Indigo ) or newer (Only for the Eclipse Version)
  48 \end{enumerate}
  49 
  50 Now for getting all those items:
  51 \begin{enumerate}
  52 \item The plugin runs on the following platforms: Windows 32/64 bit, Linux 32/64 bit and Mac OS X 64bit. Note that \textbf{Mac OS X 32 bit} is \textbf{not supported}. If you want to know why, please consult the Frequently Asked Questions section.
  53 \item Download and install Oracle/Sun's Java Version 6 (Java SE6): \href{http://java.sun.com/javase/downloads/widget/jdk6.jsp}{Download JDK}
  54 
  55 \textit{Note:} The plugin uses Java SE6, thus older versions (like 1.4 or 1.5) don't work with it.\\
  56 \textit{Note:} Please double check the java installed on your system. On some machines there is the OpenJDK or other Java versions. Just use Oracle/Sun's so there will be no problems.
  57 
  58 \item Download and install Python 2.x:
  59  \begin{enumerate}
  60    \item \textbf{Windows:} Download and install it from here: \href{http://python.org/download/}{Download Python} , selecting a 2.x version
  61    \item \textbf{Linux:} Use the distribution's package manager for installing it.
  62    \item \textbf{Mac:} Download and install it from here: \href{http://python.org/download/}{Download Python} , selecting a 2.x version
  63    \item For other operating systems, check the guide over here: \href{http://wiki.python.org/moin/BeginnersGuide/Download}{Python download and install}
  64   \end{enumerate}
  65  \textit{Note:} Please ensure you install the 2.x version. Version 3.x is \textbf{not} yet supported by the Wesnoth's WML Tools.
  66 
  67 \item You will need to have a wesnoth version 1.9.x, trunk or newer, in order for the plugin's features to correctly work.
  68 
  69 \item \textit{Note:} The following 2 steps are only neccessary if you will use the Eclipse version of the plugin.
  70 
  71 \item Download ``Eclipse" (The download links are in the right. Please ensure you are downloading at least the \textbf{3.5} version, otherwise the plugin will not work. Also, \textbf{don't} download a \textbf{64 bit version} unless you are sure you have Java JDK on 64 bit. If you are unsure, select the 32 bit version):   \href{http://www.eclipse.org/downloads/packages/eclipse-classic-37/indigor}{Download Eclipse Classic}
  72 
  73 \item Extract the downloaded archive in a known location and launch the executable (eclipse / eclipse.exe)
  74 
  75 \end{enumerate}

  77 \section{Installing the plugin}
  78 
  79 \subsection{Eclipse version}
  80 The Eclipse version, like the name says, needs a working Eclipse environment.
  81 
  82 \begin{enumerate}
  83 \item After launching Eclipse, go to the ``Help'' menu - Install new Software. Then, check the checkbox ``Contact all update sites during install to find required software" (it can be found in the bottom of the page).
  84 
  85 \begin{center}
  86 \includegraphics[scale=0.7]{install_new_software.png}
  87 \end{center}
  88 
  89 \item Insert the link \textbf{http://eclipse.wesnoth.org/updates} in the textbox in the top and press Enter. The list will be populated with some items.
  90 \item Select the single item ``Wesnoth User Made Content Plugin" and press finish.
  91 
  92 \textit{Note:} If you are prompted for any license agreements or certificates press Yes on all (if you agree).\\
  93 \textit{Note:} After the plugin has installed, press ``Restart workbench now" when you are prompted.
  94 \item Now the plugin is ready to be used.
  95 \end{enumerate}

 106 \section{Features}
 107 
 108 The plugin has the following list of abilities (not exhaustive):
 109 \begin{itemize}
 110   \item Easily creation of UMC Addons
 111   \item Quick ``Writing" of WML code by using WML Wizards, by focusing on the content
 112   \item Integrated launch of the Game or Map Editor
 113   \item Integrated launch of the WML tools on individual projects/files
 114   \item Support for multiple Wesnoth versions
 115   \item Integrated powerful WML Editor with autocomplete, go to macro definition, etc
 116 \end{itemize}

 118 \section{Using the plugin}
 119 After you have started the plugin, you can use its features.
 120 
 121 But before of all, you must setup your environment, so it knows where your wesnoth game is located. By default, after you have launched the plugin for the first time, you will be greeted with a small assistant that will help you ``setup the workspace". The preferences page for the plugin will pop up and you must fill all fields, so the plugin will work as intended. After you completed the fields, press \textbf{OK}. Then the plugin will create for convenience (i.e. if you want) wesnoth projects for each directory in your \textbf{useraddons/data/add-ons} directory. The working directory is the directory that contains the data, images, manual, sounds directories of Wesnoth.
 122 
 123 If there were no errors a message window will open saying: \textbf{Workspace was setup successfully}.
 124 
 125 \subsection{Wesnoth installations}
 126 The plugin supports working with multiple installations ( we'll call them ``installs" ). The installs can be setup from the Preferences panel. An installation is called valid if:
 127 \begin{itemize}
 128   \item It is defined in the Installs page
 129   \item It has set all paths and they point to existing objects (currently the following paths can be set: wesnoth working directory, wesnoth executable path, wesnoth user addons directory and wesnoth wml tools path)
 130 \end{itemize}
 131 
 132 The Installs Preferences Page will list all currently saved installs. You can view, edit, remove or set as default a specific installation.( \textit{Note:} You cannot modify the name of the install after creation. For that, you need to delete and add it again with another name.) The installs have also attached a Wesnoth version to them. This is for now, just for informational purposes. It is not used effectively in others area than the Preferences page.
 133 
 134 All installs have in common the same python path. That means, each installation points to the same, current Python path set.
 135 
 136 A default installation, is used at times like:
 137 \begin{itemize}
 138   \item A new project is created
 139   \item A project uses a previous existing install, which is no more valid
 140 \end{itemize}
 141 The first created install ever will become automatically default. If a default installation is deleted and there are another installs, one of them will be selected as the new default.
 142 
 143 After the user has configured the installs, he can use them on a per-project basis. Each project has a properties page ( Right click on project -$>$ Properties -$>$ Wesnoth Project ). There the user can configure project specific settings, including the install used in the context of that project.
 144 
 145 \subsection{Wizards}
 146 There are some wizards available in the plugin, that will create either projects or config files, based on the specified input. This wizards are available by going to the ``File" menu - New, and from that list selecting one of the Wesnoth wizards.
 147 
 148 There are 3 wizards categories:
 149 \begin{enumerate}
 150 \item Project wizards - ``Empty project" and ``Campaign" wizards create new projects in the workspace. The former will create a basic addon directory structure. The latter will create a campaign and its directory structure.
 151 \item File wizards - ``Wesnoth config file", ``Scenario", ``Era" and ``Faction" wizards create a new file in a selected container.
 152 \item Wizard launcher - This is a special wizard. It takes a wml tag as input, and generate subsequent needed tags and key inputs. This is generated based on the wml schema, so if the schema is incomplete some tags won't be available. This wizard can create a new file or copy the resulted WML into the current edited file.
 153 \end{enumerate}

 155 \subsection{Menus}
 156 There are currently 2 types of menus: the context menus for different file/container types and the main menu. For a better distinction the menus have the wenoth icon near each item.
 157 
 158 \begin{description}
 159 \item{\textbf{Project context menu}} - right click on the projects created with the plugin
 160 
 161    \textit{Wesnoth project report} - will show a simple report with the numer of maps, scenarios and units.\\
 162    \textit{Open campaign in game} - will start the current project's campaign (if any) in wesnoth. For this to work, you must have a ``\_main.cfg" file defined, and a campaign in it.\\
 163    \textit{Upload addon} - will upload the specified directory on the wesnoth addon server. The status will be outputed to the console.\\
 164    \textit{Regenerate build files} - recreates the ``build.xml" file. That is used if the current project is one relative to useraddons\\
 165    \textit{Builders} - utilities for adding/removing the wesnoth/xtext builders. Don't use them until you know what you are doing.
 166 
 167 \item{\textbf{Container context menus}} - right click on any container
 168 
 169     \textit{WML Tools} - provides some options for using the wmltools with the specified project.
 170 
 171 \item{\textbf{``maps" folder context menu}} - right click on the ``maps" directory
 172 
 173    \textit{Import map} - Shows a file selection window that will ley you select a .map file that will be copied in your maps directory.
 174 
 175 \item{\textbf{.cfg files context menu}} - right click on any .cfg file
 176 
 177    \textit{Open scenario in game} - opens the selected file's scenario (if it contains one) in wesnoth.\\
 178    \textit{WML Tools} - provides some options for using the wmltools with the specified file. (e.g. run wmllint against the file and see the output in the console) \\
 179    \textit{Preprocessor} - provides ways of preprocessing and showing the result in an editor inside eclipse.
 180 
 181 \item{\textbf{.map files context menu}} - right click on any .map file
 182 
 183    \textit{Open map in editor} - will open the selected map in the wesnoth map editor.
 184 
 185 \item{\textbf{Editor context menu}} - right click in the editor
 186    \textit{Validate} - will validate the entire file (this is an expensive action)
 187 
 188 \item{\textbf{Main menu}} - This is a menu near the ``Window" menu bar
 189 
 190    \textit{Open editor} - Will open the game editor with a blank map.\\
 191    \textit{Open game} - Will open the game.\\
 192    \textit{Import map} - It will import a map in selected container.\\
 193    \textit{Open editor} - Open the selected map in the game editor.\\
 194    \textit{Setup workspace} - Will setup the workspace if it's not setup. This will open the preferences page if any preferences is missing, and then it will create the wesnoth project for mainline and user addons directories if you want.\\
 195    \textit{Open plugin's preferences} - Opens the plugin preferences page.\\
 196 \end{description}

 198 \section{WML Editor}
 199 The plugin contains a full-fledged WML Editor. It recognizes the .cfg files and opens them in the WML Editor by default. The Editor supports UTF-8 files not just ASCII ones. The editor is one of the most important features of the plugin.
 200 
 201 \textit{Note:} Albeit the editor can open external files ( that is, outside the workspace ), all the cool features that come with the editor, will not be available until the file belongs to a project.
 202 
 203 \subsection{Content assist}
 204 At any time, when writing WML Code, one could activate the content assist by the \textbf{CTRL + SPACE} shortcut. A small inline popup will open showing suggestions on what to complete. Currently the plugin can auto complete:
 205 \begin{itemize}
 206   \item Scenario ids
 207   \item WML Tags
 208   \item WML Keys
 209   \item WML Macros
 210 \end{itemize}
 211 
 212 Each item has a little icon to help the user distinguish between suggestions types.
 213 
 214 \subsection{WML macros}
 215 One can view the definition of a WML macro, by hovering / going with the caret over the call, and pressing \textbf{F2}. A small popup will open showing the macros' definition. Alternatively, Pressing \textbf{F3}, will open the macro's definition file, so you can modify it or inspect related macros.

 217 \section{Use cases}
 218 \subsection{Import an already existing wesnoth addon into the plugin}
 219 For this you need just 2 single steps:
 220 \begin{enumerate}
 221 \item Open eclipse. Click the ``New" main menu bar - Project. Select the ``Empty project" from ``Wesnoth" category.
 222 \item Enter the name of the project. Uncheck ``Use default location", and press ``Browse...". Navigate to the addon's directory and then continue the wizard as necessary.
 223 \end{enumerate}

 225 \subsection{Updating the plugin}
 226 The plugin, due to Eclipse's architecture, can be updated very simple. At any time, the user can check for new updates, by going to the menu: Help-$>$ Check for Updates. If there are any new versions released, they will get downloaded and applied. This way you'll need to download just the bits updated, not the whole eclipse or standalone version.

 228 \section{Frequently Asked Questions}
 229 
 230 \subsection{What is this plugin all about?}
 231 The plugin is in fact something like a WML IDE ( \href{http://en.wikipedia.org/wiki/Integrated_development_environment}{Wikipedia - IDE}). Basically it offers you some features that greatly help anybody who makes User Made Content in WML.
 232 
 233 \begin{description}
 234 \item - It has wizards for new campaigns, scenarios, factions, eras, and everything else you might think of. You just complete the needed values and it generates the wml (with nice indentation).
 235 \item - It has front-ends for launching wmltools (like wmllint, wmlscope, wmlindent on files/directories), the wesnoth game, and the map editor.
 236 \item - It has a specialized WML editor that has syntax highlighting, tag folding and outline, autocompletion, navigation to macro definitions and so on.
 237 \item - It has integrated addon management features that allow to upload or download arbitrary addons. 
 238 \end{description}
 239 
 240 \subsection{The IDE doesn't start anymore after update}
 241 If you have just used the auto-update mechanism, and after the restart the IDE doesn't start anymore, then you can use a simple script to fix it. Just go to the download page (\url{http://eclipse.wesnoth.org/download.html}) and download the "Post-Update fix" script for your Operating System and execute it. Place it on the same level with the UMC IDE binary (wesnoth\_umc\_dev), and run it. Now the IDE should start fine.

 246 \subsection{Where I can find the logs?}
 247 If you encounter any errors during the usage of the plugin, the plugin logs them in the: \textbf{$<$your temporary directory$>$/wesnoth\_plugin/logs}.
 248 That is usually on linux:\\
 249 \indent \textbf{/tmp/wesnoth\_plugin/logs}\\
 250 or on windows:\\
 251 \indent \textbf{C:/Documents and settings/$<$your username$>$/Local Settings/Temp/wesnoth\_plugin/logs} \\
 252 or\\
 253 \indent \textbf{C:/Users/$<$your username$>$/Local Settings/Temp/wesnoth\_plugin/logs}
 254 
 255 \subsection{Why Mac OS X 32 bit is not supported?}
 256 Apple decided to manage itself the versions that can run on each Mac version. Thus the modern Java SE6 version - which is used by the plugin - is not available on 32 bit Macs but on 64 bit only. The 32 bit Macs have just Java SE5 version available. Trying to port the plugin to the ``outdated'' java version is not likely to be done, so please consider upgrading your OS.
