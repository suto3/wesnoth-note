  95  * Helper class to do a runtime test whether an event is in a set.
  96  *
  97  * The class is supposed to be used in combination with find function. This
  98  * function is used in the fire functions to make sure an event is send to the
  99  * proper handler. If not there will be a run-time assertion failure. This
 100  * makes developing and testing the code easier, a wrong handler terminates
 101  * Wesnoth instead of silently not working.

 106         /**
 107          * If found we get executed to set the result.
 108          *
 109          * Since we need to return true if found we always return true.
 110          */

 113         {
 114                 return true;
 115         }

 118 bool dispatcher::fire(const ui_event event, widget& target)
 119 {
 120         assert(find<set_event>(event, event_in_set()));
 121         switch(event) {
 122                 case LEFT_BUTTON_DOUBLE_CLICK:
 123                         return fire_event_double_click<LEFT_BUTTON_CLICK, LEFT_BUTTON_DOUBLE_CLICK,
 124                                         &event_executor::wants_mouse_left_double_click, signal_function>(this, &target);
 125 
 126                 case MIDDLE_BUTTON_DOUBLE_CLICK:
 127                         return fire_event_double_click<MIDDLE_BUTTON_CLICK, MIDDLE_BUTTON_DOUBLE_CLICK,
 128                                         &event_executor::wants_mouse_middle_double_click, signal_function>(this, &target);
 129 
 130                 case RIGHT_BUTTON_DOUBLE_CLICK:
 131                         return fire_event_double_click<RIGHT_BUTTON_CLICK, RIGHT_BUTTON_DOUBLE_CLICK,
 132                                         &event_executor::wants_mouse_right_double_click, signal_function>(this, &target);
 133 
 134                 default:
 135                         return fire_event<signal_function>(event, this, &target);
 136         }
 137 }

 200 /**
 201  * @page event_dispatching Event dispatching.
 202  *
 203  * @section introduction Introduction
 204  *
 205  * This page describes how the new event handling system works, since the
 206  * system is still work in progress it might be out of date with the actual
 207  * code. It also contains some ideas that might change later on. Some parts are
 208  * explained in the interface and will be integrated in this document later.
 209  *
 210  * Since the event handling code hasn't been cast in stone yet some scenarios
 211  * for solving the problem are discussed first and then the solution that is
 212  * chosen in more detail.
 213  *
 214  * After SDL has generated and event it needs to be turned into an event which
 215  * the widgets can use.
 216  *
 217  * @section handling_solution The implementation solutions.
 218  *
 219  * For the event handling we use a few use case scenarios and show the possible
 220  * solutions.
 221  *
 222  * @subsection sample The sample window
 223  *
 224  * In our samples we use this sample window with the following components:
 225  * - a window W
 226  * - a container C
 227  * - a button B
 228  *
 229  * These are arranged accordingly:
 230  * @code
 231  *
 232  *   ---------------------
 233  *  |W                     |
 234  *  |                      |
 235  *  |  -----------------   |
 236  *  | |C              |^|  |
 237  *  | |               |-|  |
 238  *  | |  ----------   |#|  |
 239  *  | | |B         |  | |  |
 240  *  | |  ----------   | |  |
 241  *  | |               |-|  |
 242  *  | |               |v|  |
 243  *  |  -----------------   |
 244  *  |                      |
 245  *   ---------------------
 246  *
 247  * @endcode
 248  *
 249  * @subsection scenarios Possible scenarios
 250  *
 251  * The scenarios are:
 252  * - An event that is wanted by none.
 253  * - A mouse down event that should focus C and set the pressed state in B.
 254  * - A mouse wheel event, which first should be offered to B and if not handled
 255  *   by B should be handled by C.
 256  *
 257  * @subsection all_queues Pass the event through all queues
 258  *
 259  * In this solution the event will be passed through all possible queues and
 260  * tries sees where the event sticks. This following sections describe how the
 261  * events are tried for this usage scenario.
 262  *
 263  * @subsubsection unhandled Unhandled event
 264  *
 265  * - W pre child
 266  * - C pre child
 267  * - B pre child
 268  * - W child
 269  * - C child
 270  * - B child
 271  * - W post child
 272  * - C post child
 273  * - B post child
 274  *
 275  * @subsubsection mouse_down Mouse down
 276  *
 277  * - W pre child
 278  * - C pre child -> set focus -> !handled
 279  * - B pre child -> set pressed state -> handled
 280  *
 281  * @subsubsection mouse_wheel Mouse wheel
 282  *
 283  * - W pre child
 284  * - C pre child
 285  * - B pre child -> We can't scroll so ignore
 286  * - W child
 287  * - C child
 288  * - B child
 289  * - W post child
 290  * - C post child -> Scroll -> handled
 291  *
 292  * @subsection chain Pass the events in a chain like fashion
 293  *
 294  * In this solution the events are send to the pre- and post queue of all but
 295  * the last possible widget and to the child of the last widget. The pre queue
 296  * will be send from top to bottom, the post queue from bottom to top.
 297  *
 298  * @subsubsection unhandled Unhandled event
 299  *
 300  * - W pre child
 301  * - C pre child
 302  * - B child
 303  * - C post child
 304  * - W post child
 305  *
 306  * @subsubsection mouse_down Mouse down
 307  *
 308  * - W pre child
 309  * - C pre child -> set focus -> !handled
 310  * - B child -> set pressed state -> handled
 311  *
 312  * @subsubsection mouse_wheel Mouse wheel
 313  *
 314  * - W pre child
 315  * - C pre child
 316  * - B child -> We can't scroll so ignore
 317  * - C post child -> Scroll -> handled
 318  *
 319  * @subsection evaluation Evaluation
 320  *
 321  * When using the first solution it's possible to drop the child queue since
 322  * everything falls in pre or post. But there is a scenario that's a bit ugly
 323  * to solve with the first solution:
 324  *
 325  * Assume there is a listbox with toggle panels and on the panel there are a
 326  * few buttons, the wanted behavior is:
 327  * - if clicked on the panel it should toggle, which may or may not be allowed.
 328  * - if clicked on a button in the panel, we want to make sure the panel is
 329  *   selected, which again may or may not be allowed.
 330  *
 331  * With solution 2 it's rather easy:
 332  *
 333  * Click on panel:
 334  * - W pre child
 335  * - C child -> Test whether we can toggle -> handled, halt = !toggled
 336  *
 337  * Click on button in panel:
 338  * - W pre child
 339  * - C pre child -> Test whether we can select -> handled = halt = !selected
 340  * - B child -> do button stuff -> handled
 341  *
 342  * Since for the different clicks, different queues are triggered it's easy to
 343  * add a different handler there.
 344  *
 345  * With solution 1:
 346  *
 347  * Click on panel:
 348  * - W pre child
 349  * - C pre child -> handler 1 -> if last in queue -> solution 2 C child
 350  *
 351  * Click on button in panel:
 352  * - W pre child
 353  * - C pre child -> handler 2 -> if !last in queue -> solution 2 C pre child
 354  * - B pre child -> do button stuff -> handled
 355  *
 356  * Not that different from solution 2, the two handlers are installed in the C
 357  * pre event. But we need to manually check whether we're really the last,
 358  * which means the code to check whether there are more handlers at a lower
 359  * level is needed for both solutions. In solution 1 this test needs to be done
 360  * twice versus once in solution 2. Also the fact that the queues for the
 361  * events are processed in reverse order on the way back sounds more
 362  * initiative.
 363  *
 364  * @section processing_raw_events Processing the raw events.
 365  *
 366  * This section describes how the events generated by SDL are send as our own
 367  * events to the various widgets. The first step in sending an event is to
 368  * decode it and send it to a registered dispatcher.
 369  *
 370  * - gui2::event::sdl_event_handler handles the SDL events.
 371  * - gui2::event::dispatcher has the registered dispatchers.
 372  *
 373  * In general a dispatcher is a window which then needs to send this event to
 374  * the widgets. The dispatcher is just a simple part which fires events and
 375  * finds a handler for the event. This is not to the liking of most widgets,
 376  * they don't want to handle raw events but get a polished and clean event. No
 377  * button up and down and then try to figure out whether it needs to act as if
 378  * it was clicked upon, no simply op and down to change the appearance and a
 379  * click event to do the clicking actions. And don't even try to convince a
 380  * widget to determine whether this up event was a single or double click.
 381  * Widgets like to sleep with nice dreams and not having nightmares where SDL
 382  * events haunt them.
 383  *
 384  * In order to remedy that problem there's the gui2::event::distributor
 385  * class, it's the class to do the dirty job of converting the raw event into
 386  * these nice polished events. The distributor is in general linked to a window,
 387  * but a widget can install it's own distributor if it needs to know more of the
 388  * raw events as still left in the polished events. At the time of this writing
 389  * no widget needs this feature, but the toggle panel might need it.
 390  *
 391  * After the distributor has polished the event and send it on its way to the
 392  * widget the dispatcher needs to make sure the event is properly dispatched to
 393  * the widget in question and also notify its parents by means of the previously
 394  * described event chain.
 395  *
 396  * @subsection sdl_event Get the SDL events
 397  *
 398  * The first step in event handling is getting the events in the first place.
 399  * Events are generated by SDL and placed in a queue. The Wesnoth code processes
 400  * this queue and thus handles the events. The part which does the first
 401  * handling isn't described here since it's (secretly) intended to be replaced
 402  * by the @ref gui2::event::sdl_event_handler class. Instead we directly jump to this
 403  * class and explain what it does.
 404  *
 405  * The main handling function is @ref gui2::event::sdl_event_handler::handle_event which
 406  * as no real surprise handles the events. The function is a simple multiplexer
 407  * which lets other subfunctions to the handling of specific events.
 408  *
 409  * @todo Describe drawing and resizing once the code is stable and working as
 410  * wanted in these areas.
 411  *
 412  * @subsubsection handler_mouse Mouse motion events
 413  *
 414  * If a dispatcher has captured the mouse it gets the event, no questions asked.
 415  * If not it goes through all dispatchers and finds the first one willing to
 416  * accept the mouse event.
 417  *
 418  * This means a mouse event is send to one dispatcher.
 419  *
 420  * @subsubsection handler_mouse_button_down Mouse button down events
 421  *
 422  * Turning the mouse wheel on a mouse generates both an down and up event. It
 423  * has been decided to handle the wheel event in the button up code so wheel
 424  * events are here directly dropped on the floor and forgotten.
 425  *
 426  * The other buttons are handled as if they're normal mouse events but are
 427  * decoded per button so instead of a button_down(id) you get button_down_id.
 428  *
 429  * @subsubsection handler_mouse_button_up Mouse button up events
 430  *
 431  * The mouse wheel event is handled as if it's a keyboard event and like the
 432  * button_down they are send as wheel_id events.
 433  *
 434  * The other mouse buttons are handled the same as the down buttons.
 435  *
 436  * @subsubsection handler_keyboard Keyboard events
 437  *
 438  * There are three types of keyboard events, the already mentioned mouse wheel
 439  * events, the key down and key up event. When a key is pressed for a longer
 440  * time several key down events are generated and only one key up, this means
 441  * the key up is rather useless. Guess what, the multiplexer already drops that
 442  * event so we never get it.
 443  *
 444  * If the keyboard event is a mouse wheel event it's directly send to the
 445  * dispachting queue; either the dispatcher that captured the keyboard or the
 446  * last dispatcher in the queue.
 447  *
 448  * If the event is a real keyboard action it's first tried as hotkey. In order
 449  * to do so the target dispatcher is first determined, either the dispatcher
 450  * that captured the keyboard or the last dispatcher in the queue. Then it's
 451  * tried whether a hotkey and whether the hotkey can be processed. If the
 452  * hotkey isn't processed the keyboard event is send to the dispatcher as
 453  * normal keyboard event.
 454  *
 455  * The hotkey processing will have several queues (to be implemented in 1.9):
 456  * - global hotkeys that always work eg toggling fullscreen mode.
 457  * - main screen hotkeys, these work when one of the dialogs is shown without
 458  *   other dialogs on top of them. These hotkeys are for example
 459  *   preferences. The main screens are:
 460  *   - title screen
 461  *   - game
 462  *   - editor
 463  *   - mp lobby
 464  * - map screen hotkeys, these work when a map is shown eg toggle grid. The
 465  *   screens are:
 466  *   - game
 467  *   - editor
 468  * - local hotkeys, these are hotkeys that only work in a specific dialog eg
 469  *   recruit unit only works in the game screen.
 470  *
 471  * The queues are processed in from bottom to top in the list above, this
 472  * allows an item to use a hotkey but have another handler function. Eg
 473  * preferences in the editor might open another preferences dialog.
 474  *
 475  * @todo The hotkeys need to be implemented like above in 1.9.
 476  *
 477  * @todo This might change in the near future.
 478  *
 479  * @subsection distributor Event polishing and distribution
 480  *
 481  * The event distributor has the job to find the widget that should receive the
 482  * event and which event(s) to send from a single event. In general an event is
 483  * first send to the widget as-is, sending the raw events allows other
 484  * distributors to be nested between this distributor and the intended target
 485  * widget. Or the intended widget might not really be the intended widget but
 486  * another distributor that wants to dispatch the event internally.
 487  *
 488  * However in the common cases this raw event isn't handled and the distributor
 489  * needs to send the polished events. In the following sections the details of
 490  * the conversion from raw to polished is described, it intentionally lacks the
 491  * part of sending the raw events as well since it adds no value.
 492  *
 493  * A widget can capture the mouse, which means all mouse events are send to this
 494  * widget, regardless where the mouse is. This is normally done in a mouse down
 495  * event (for a button) so all events following it are send to that widget.
 496  *
 497  * @subsection mouse_motion Mouse motion
 498  *
 499  * This section describes the conversion from a raw mouse motion to the polished
 500  * events it can generate:
 501  * - @ref gui2::event::MOUSE_ENTER "MOUSE_ENTER"
 502  * - @ref gui2::event::MOUSE_LEAVE "MOUSE_LEAVE"
 503  * - @ref gui2::event::MOUSE_MOTION "MOUSE_MOTION"
 504  *
 505  * When the mouse is captured that widget will only receive motion events.
 506  *
 507  * If not captured the code checks whether the widget underneath the mouse is
 508  * the same widget as at the last motion if event. If so that widget gets a
 509  * motion event.
 510  * If not the widget that before was underneath the mouse pointer (if any) gets
 511  * a leave event and the widget newly underneath the mouse pointer (if any) gets
 512  * an enter event.
 513  *
 514  * @subsection mouse_button Mouse buttons
 515  *
 516  * The mouse button code is a bit more complex and is separated in the various
 517  * events to be send.
 518  *
 519  * @subsubsection mouse_button_down Mouse button down
 520  *
 521  * Some things start simple, so does the event of pressing down a mouse button.
 522  * All it does is send the event to the widget as one of the following events:
 523  * - @ref gui2::event::LEFT_BUTTON_DOWN "LEFT_BUTTON_DOWN"
 524  * - @ref gui2::event::MIDDLE_BUTTON_DOWN "MIDDLE_BUTTON_DOWN"
 525  * - @ref gui2::event::RIGHT_BUTTON_DOWN "RIGHT_BUTTON_DOWN"
 526  *
 527  * @todo Validate the code it seems a down event with a captured mouse doesn't
 528  * really work as wanted. (Rare case but should work properly.) In general the
 529  * mouse event handling needs testing to see whether the proper events are send
 530  * all the time.
 531  *
 532  * @subsubsection mouse_button_up Mouse button up
 533  *
 534  * Simplicity ends here.
 535  *
 536  * @todo Document further.
 537  *
 538  * @subsubsection mouse_click Mouse click
 539  *
 540  * So the button up event has asked for mouse click, now we need to test whether
 541  * the click will be a click or a double click. A double click is generated when
 542  * the same widget is clicked twice in a short time and causes the following
 543  * events:
 544  * - @ref gui2::event::LEFT_BUTTON_DOUBLE_CLICK "LEFT_BUTTON_DOUBLE_CLICK"
 545  * - @ref gui2::event::MIDDLE_BUTTON_DOUBLE_CLICK "MIDDLE_BUTTON_DOUBLE_CLICK"
 546  * - @ref gui2::event::RIGHT_BUTTON_DOUBLE_CLICK "RIGHT_BUTTON_DOUBLE_CLICK"
 547  *
 548  * Otherwise one of the following single clicks is generated:
 549  * - @ref gui2::event::LEFT_BUTTON_CLICK "LEFT_BUTTON_CLICK"
 550  * - @ref gui2::event::MIDDLE_BUTTON_CLICK "MIDDLE_BUTTON_CLICK"
 551  * - @ref gui2::event::RIGHT_BUTTON_CLICK "RIGHT_BUTTON_CLICK"
 552  *
 553  * @subsubsection double_click To double click or not to double click
 554  *
 555  * Wait a second, a widget has a field whether or not it wants a double click
 556  * for a certain mouse button and now I see that it's bluntly ignored by the
 557  * distributor. Indeed the distributor doesn't care about what the widget wants,
 558  * it does what it wants and leaves the sorting out what's wanted to the
 559  * dispatcher.
 560  *
 561  * The problem is that in the chain events are send to one widget that may not
 562  * be interested in a double click, but another widget in the chain is. There
 563  * are several solutions to this problem:
 564  * -# Sending a click followed by a double click.
 565  * -# Sending a click with a tag field that it actually is a double click.
 566  * -# Send a double click and turn it into a click if the double click is
 567  *    unwanted.
 568  *
 569  * The first solution has the disadvantage that a toggle panel likes a click and
 570  * double click, the first click selects the second deselects and now the
 571  * deselected panel gets a double click. When the panel now checks whether it's
 572  * selected it's not and might take the wrong action upon it.
 573  *
 574  * The second option is possible but would be rather intrusive in the code,
 575  * since it would generate another event signature. Adding a signature just for
 576  * this special case seemed a bit too much effort vs. gain. Also the widget
 577  * needs to check whether a click is a click or a double click and choose a
 578  * different code path for it. This in turn would mean a signal handler
 579  * secretly might handle two events and lowers the transparency of the code.
 580  *
 581  * The third option also adds some special case handling but the scope is
 582  * limited and only one part knows about the tricks done.
 583  *
 584  * The last option has been chosen and the dispatcher build the event chain and
 585  * while building the chain it looks whether the widget wants the double click
 586  * or not. It does this test by looking at the wants double click function and
 587  * not test for a handler. The double click test function is made for this
 588  * purpose and depending on the handler might again do the wrong thing.
 589  * (A certain toggle panel might not want to do something on a double click but
 590  * also not being deselected upon a double click. The latter to keep the UI
 591  * consistent, a double click on a toggle panel might execute a special function
 592  * or not, but always keep the panel selected. (That is if the panel can be
 593  * selected.))
 594  */
